# 핀로그 콘텐츠 팩트체크 & 검증

핀로그의 지식은 전부 `index.html` 안 `KB`/`FAQS` 배열에 인라인되어 있고, 자동 테스트가 없다.
데이터를 편집하면 개수·필드·타입·사실관계가 조용히 어긋나기 쉽다. 이 스킬은 그 회귀를 막는다.

## 1. 팩트체크 원칙 (금융 데이터의 정확성)

- **수치엔 기준일·단위를 명시한다.** "코스피 2,800" 대신 "2026-07 기준 코스피 약 2,800p"처럼.
  값이 시간에 따라 변하는 정보(지수·금리·환율·시총·법 시행일)는 **단정하지 말고** 기준시점 라벨을 붙인다.
- **변동 가능한 사실은 웹으로 재검증한다.** 제도 시행 여부, 최신 규제, 사건 진행 상황은
  기억에 의존하지 말고 공식 출처(금융위·한국은행·거래소·법령·1차 보도)를 우선 확인한다.
- **출처 없는 통계·단정 표현을 넣지 않는다.** 불확실하면 "~로 알려져 있다/논의 중"처럼 표시하고,
  상충하는 출처가 있으면 양쪽을 함께 적는다.
- **중립성.** 마스코트(오르소·고마) 톤은 유지하되, 쟁점(공매도·노란봉투법 등)은 한쪽으로 단정하지 않는다.

## 2. 데이터 구조 규약 (검사기가 강제하는 실제 스키마)

`index.html` 파싱으로 확인된 실측 기준(2026-07):

- **개수**: KB **108** = 일반용어 **87** + 학습주제 **21**, FAQ **63**
- **주제 판별은 반드시 `topic === true`** 로 한다. `category === '핀로그 학습주제'` 는 **11개뿐**이라
  판별 기준이 될 수 없다(carbon-tax·short-selling 등 10개 주제는 실제 카테고리를 가짐).
- **KB 공통 필드(108개 전부)** — 타입 고정:
  - `id`(str, 고유), `term`(str), `category`(str, 아래 10종), `short`(str), `detail`(str),
    `example`(str), `keywords`(**배열**, 검색 별칭 충분히)
- **학습주제 전용 필드(topic===true 21개)**:
  - `topic`(**boolean true**), `weeks`(str), `title`·`conclusion`·`interview_tip`(str),
    `hashtags`·`issues`(**배열**), `faqs`(**배열**, 원소는 `{question, answer}`)
- **의도된 빈 값**: `id` 가 `t_` 로 시작하는 순수 학습주제 11개(= `category==='핀로그 학습주제'`)는
  `detail`·`example` 이 **비어 있는 게 정상**이다. 이건 오류가 아니다.
- **카테고리 10종**: 시장/지수, 거시경제, 주식/채권, 파생상품, 재무/회계, 규제/제도,
  디지털금융, 사건사례, 취업/직무, 핀로그 학습주제
- **FAQ 필드**: `q`, `a`, `term`, `weeks`, `cat` (모두 str). `term` 은 가급적 KB 용어/키워드와 맞춘다.
- **XSS**: 어떤 데이터도 신뢰 입력이 아니다. 렌더 경로는 반드시 `escapeHtml` 을 거친다(기존 규칙).

## 3. 편집 워크플로 (반드시 이 순서)

1. **수정 전 스냅샷**: 현재 개수·구조를 파악한다.
   ```bash
   node scripts/validate-content.js
   ```
2. **편집**: `KB`/`FAQS` 를 위 구조 규약대로 수정한다. 배열은 배열로, `topic` 은 boolean 으로.
3. **검증(필수)**: 다시 실행해 **오류 0건**을 확인한다. 오류가 있으면 완료가 아니다.
   ```bash
   node scripts/validate-content.js
   ```
4. **수치 동기화**: 용어/주제/FAQ 개수가 바뀌었으면 `README.md`·`package.json`·`index.html`의
   `og:description`·`CLAUDE.md` 수치를 함께 고친다. 검사기의 "문서 정합성" 경고가 사라져야 한다.
5. **눈 검증**: 브라우저로 `index.html` 을 열어 새 항목의 검색·칩·렌더가 정상인지 확인한다.

## 4. 완료 선언 규칙 (Verification Before Completion)

- **최신 실행 결과 없이 "완료"라고 말하지 않는다.** 코드/데이터만 보고 됐다고 판단하지 말 것.
- 완료 보고에는 반드시 **`node scripts/validate-content.js` 의 최신 통과 출력**(개수·경고 0)을 근거로 포함한다.
- 수정 전/후 개수·구조 변화를 한 줄로 비교해 보고한다(예: "용어 87→88, FAQ 63 유지, 검사 통과").

## 5. 검사기가 잡는 것 / 못 잡는 것

- **잡는다(결정적)**: 개수 불일치, 중복·빈 `id`, 필드 누락·타입 오류, 빈 필수값, 미정의 카테고리,
  `topic` 비-boolean, 빈 배열 필드, FAQ 필드 결함, 문서 수치 드리프트(경고), 깨진 FAQ 참조(경고).
- **못 잡는다(사람·웹 확인 몫)**: 사실관계의 진위, 수치의 최신성, 설명의 중립성·품질.
  → 그래서 1절 팩트체크 원칙이 검사기와 별개로 항상 필요하다.
